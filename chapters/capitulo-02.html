Introducción

¿Qué es ROS 2?

ROS 2 (Robot Operating System 2) es un framework de código abierto para el desarrollo de software robótico. No es un sistema operativo en el sentido tradicional, sino un conjunto de bibliotecas, herramientas y convenciones que simplifican la creación de aplicaciones robóticas complejas y robustas. ROS 2 fue diseñado desde cero para superar las limitaciones de ROS 1, incorporando soporte nativo para sistemas en tiempo real, comunicaciones seguras mediante DDS (Data Distribution Service), y compatibilidad con múltiples plataformas.

La versión Jazzy Jalisco, lanzada el 23 de mayo de 2024, es una distribución LTS (Long Term Support) con soporte hasta mayo de 2029. Está basada en Ubuntu 24.04 LTS (Noble Numbat) y representa la opción más estable y recomendada para proyectos de producción y formación académica.

¿Por qué instalar ROS sobre Ubuntu?

Ubuntu es la plataforma oficialmente soportada por Open Robotics para ROS 2. Cada distribución de ROS 2 se vincula con una versión específica de Ubuntu: Jazzy Jalisco requiere Ubuntu 24.04 LTS. Instalar ROS 2 sobre Ubuntu garantiza compatibilidad completa con todos los paquetes binarios disponibles en los repositorios oficiales, evita problemas de dependencias y proporciona acceso a la documentación y soporte comunitario más extenso.

¿Por qué utilizar WSL2?

WSL2 (Windows Subsystem for Linux 2) permite ejecutar una distribución de Linux directamente sobre Windows sin necesidad de una máquina virtual tradicional. WSL2 utiliza un kernel Linux real proporcionado por Microsoft, ejecutado sobre el hipervisor Hyper-V. Esto significa que se obtiene compatibilidad total con las llamadas al sistema Linux, mientras se mantiene la integración con el entorno Windows.

Desde la versión WSL 2.0, Microsoft incluye WSLg (Windows Subsystem for Linux GUI), que permite ejecutar aplicaciones gráficas de Linux —como RViz2 y Gazebo— directamente en el escritorio de Windows, sin necesidad de configurar un servidor X externo.

Virtualización requerida

WSL2 necesita que la virtualización por hardware esté habilitada en la BIOS/UEFI. En procesadores Intel, busque Intel VT-x; en AMD, busque AMD-V (SVM). Sin esta opción activa, la instalación de WSL2 fallará.

Paso 1

Instalar WSL2

¿Qué es WSL2?

WSL2 (Windows Subsystem for Linux 2) es la segunda generación del subsistema de Windows para Linux. A diferencia de WSL1, que traducía las llamadas al sistema Linux a llamadas NT, WSL2 ejecuta un kernel Linux real dentro de una máquina virtual ligera gestionada por Hyper-V. Esto proporciona compatibilidad completa con las llamadas al sistema Linux, rendimiento de E/S significativamente superior y la capacidad de ejecutar cualquier binario ELF de Linux.

Instalación

Abra PowerShell como Administrador. Para ello, haga clic derecho sobre el botón Inicio de Windows y seleccione "Terminal (Administrador)" o busque "PowerShell" en el menú Inicio, haga clic derecho y seleccione "Ejecutar como administrador".

Ejecute el siguiente comando:

powershell (admin) 
wsl --install

Este comando realiza automáticamente las siguientes acciones:

  • Habilita la característica opcional Virtual Machine Platform
  • Habilita la característica opcional Windows Subsystem for Linux
  • Descarga e instala el kernel Linux más reciente para WSL2
  • Establece WSL2 como la versión predeterminada
  • Descarga e instala Ubuntu como la distribución predeterminada
Reinicio obligatorio

Después de ejecutar wsl --install, Windows solicitará un reinicio del sistema. Este reinicio es obligatorio para que los componentes de virtualización se activen correctamente. No omita este paso.

Verificar la instalación

Una vez reiniciado el sistema, abra nuevamente PowerShell y ejecute:

powershell 
wsl --version

La salida mostrará información similar a:

salida

Versión DXCore: 10.0.26100.1-240331-1435.ge-release
Versión de Windows: 10.0.22631.4890

Descripción de cada campo:

Versión de kernelVersión del kernel Linux que ejecuta WSL2, actualizado por Windows Update.

Versión de WindowsBuild de Windows del sistema host. Debe ser 19041 o superior para WSL2.
Paso 2

Instalar Ubuntu 24.04

Descargar Ubuntu desde Microsoft Store

Si el comando wsl --install no instaló Ubuntu automáticamente, puede hacerlo manualmente. Abra Microsoft Store, busque "Ubuntu 24.04 LTS" y haga clic en "Obtener". También puede instalarla desde PowerShell:

powershell 
wsl --install -d Ubuntu-24.04

Crear usuario y contraseña

La primera vez que inicie Ubuntu, el sistema le solicitará crear un nombre de usuario y una contraseña. Este usuario será su cuenta principal dentro del entorno Linux y tendrá permisos sudo (superusuario) por defecto.

Consejo

Utilice un nombre de usuario corto y en minúsculas, sin espacios ni caracteres especiales. Por ejemplo: robotica, ros, dev. Este nombre aparecerá en el prompt de la terminal y en las rutas del sistema.

Actualizar el sistema

Antes de instalar cualquier software adicional, actualice todos los paquetes del sistema a sus últimas versiones:

bash 
sudo apt update
sudo apt upgrade -y
sudo apt autoremove -y

Descripción de cada comando:

  • sudo apt update — Actualiza la lista de paquetes disponibles desde los repositorios configurados.
  • sudo apt upgrade -y — Instala las versiones más recientes de todos los paquetes instalados. La opción -y acepta automáticamente la confirmación.
  • sudo apt autoremove -y — Elimina paquetes que fueron instalados como dependencias y ya no son necesarios.

Paso 5

Configurar Locale

Verifique la configuración regional actual del sistema:

bash 
locale

Si el resultado muestra LANG=C.UTF-8 o LANG=en_US.UTF-8, la configuración es correcta. Si muestra POSIX o no incluye UTF-8, ejecute los siguientes comandos:

bash 
sudo locale-gen en_US en_US.UTF-8
sudo update-locale LC_ALL=en_US.UTF-8 LANG=en_US.UTF-8
export LANG=en_US.UTF-8

¿Por qué ROS 2 requiere UTF-8?

ROS 2 utiliza UTF-8 como codificación de caracteres por defecto en todo su middleware de comunicación (DDS). Los nombres de tópicos, servicios, parámetros y mensajes deben estar codificados en UTF-8 para garantizar la interoperabilidad entre nodos escritos en diferentes lenguajes (Python, C++). Si el locale del sistema no está configurado correctamente en UTF-8, pueden producirse errores de codificación al publicar o suscribirse a mensajes que contengan caracteres no ASCII, y algunas herramientas de introspección como ros2 topic echo pueden fallar silenciosamente.

Paso 6

Agregar el repositorio oficial de ROS 2

Habilitar el repositorio Universe

El repositorio Universe contiene software de código abierto mantenido por la comunidad de Ubuntu, incluyendo dependencias necesarias para ROS 2:

bash 
sudo add-apt-repository universe

Agregar la llave GPG de ROS 2

La llave GPG permite a apt verificar la autenticidad e integridad de los paquetes descargados del repositorio de ROS 2. Sin esta llave, apt rechazará la instalación de paquetes del repositorio:

bash 
sudo curl -sSL https://raw.githubusercontent.com/ros/rosdistro/master/ros.key \
  -o /usr/share/keyrings/ros-archive-keyring.gpg

Descripción de las opciones de curl:

  • -s — Modo silencioso, no muestra la barra de progreso.
  • -S — En combinación con -s, muestra errores si ocurren.
  • -L — Sigue redirecciones HTTP.
  • -o — Guarda la salida en el archivo especificado.

Agregar el repositorio de ROS 2

Este comando crea un archivo de fuentes para apt que apunta al repositorio oficial de ROS 2 para la versión correcta de Ubuntu:

bash 
echo "deb [signed-by=/usr/share/keyrings/ros-archive-keyring.gpg] \
  http://packages.ros.org/ros2/ubuntu \
  $(. /etc/os-release && echo $UBUNTU_CODENAME) main" | \
  sudo tee /etc/apt/sources.list.d/ros2.list

Desglose del comando:

  • deb [signed-by=...] — Indica que los paquetes deben estar firmados con la llave GPG especificada.
  • http://packages.ros.org/ros2/ubuntu — URL del repositorio oficial de paquetes de ROS 2.
  • $(. /etc/os-release && echo $UBUNTU_CODENAME) — Sustituye dinámicamente el codename de Ubuntu (noble para 24.04).
  • main — Componente del repositorio que contiene los paquetes principales.
  • sudo tee — Escribe la salida en el archivo con permisos de superusuario.

Actualizar la lista de paquetes

bash 
sudo apt update

Después de ejecutar este comando, apt descargará el índice de paquetes del nuevo repositorio de ROS 2. Ahora los paquetes de ROS 2 Jazzy están disponibles para instalación.

Paso 7

Instalar ROS 2 Jazzy

bash 
sudo apt install ros-jazzy-desktop -y
Tiempo de descarga

La instalación de ros-jazzy-desktop descarga aproximadamente 1.5 GB de paquetes. El tiempo total dependerá de la velocidad de su conexión a internet. En una conexión de 100 Mbps, puede tardar entre 5 y 15 minutos.

En este curso utilizamos ros-jazzy-desktop porque incluye RViz2 y las herramientas gráficas de desarrollo necesarias para la formación. Para un robot real en producción, ros-jazzy-ros-base sería suficiente. Gazebo se instala de forma independiente en el Paso 15.

Paso 8

Configurar ROS

Agregar el source al .bashrc

Para que ROS 2 esté disponible automáticamente cada vez que abra una terminal, agregue el archivo de configuración (setup script) al archivo ~/.bashrc:

bash 
echo "source /opt/ros/jazzy/setup.bash" >> ~/.bashrc

Recargar la configuración

bash 
source ~/.bashrc

Verificar la distribución activa

bash 
echo $ROS_DISTRO

El resultado debe ser:

salida 
jazzy

¿Qué es setup.bash?

El archivo /opt/ros/jazzy/setup.bash es un script de configuración de entorno que realiza las siguientes acciones al ejecutarse:

  • Establece la variable ROS_DISTRO=jazzy, que identifica la distribución activa.
  • Agrega los ejecutables de ROS 2 al PATH del sistema (/opt/ros/jazzy/bin).
  • Configura las rutas de bibliotecas (LD_LIBRARY_PATH) para que el sistema encuentre las bibliotecas compartidas de ROS 2.
  • Configura las rutas de Python (PYTHONPATH) para que los módulos de ROS 2 sean importables.
  • Configura la variable AMENT_PREFIX_PATH, utilizada por el sistema de compilación ament para localizar paquetes instalados.
  • Habilita el autocompletado de comandos ros2 en la terminal.
Precaución

Use >> (append) y no > (overwrite) al modificar .bashrc. Un solo > sobrescribiría todo el contenido del archivo, lo que podría causar problemas graves en su configuración de terminal.

Paso 9

Inicializar rosdep

Ejecute el siguiente comando para inicializar rosdep:

bash 
sudo rosdep init
Error esperado: default sources list already exists

Si al ejecutar sudo rosdep init obtiene el siguiente error:

ERROR: default sources list file already exists:
	/etc/ros/rosdep/sources.list.d/20-default.list
Please delete if you wish to re-initialize

Este no es un error crítico. Significa que rosdep ya fue inicializado previamente, posiblemente durante la instalación del paquete python3-rosdep. El archivo de fuentes ya existe y no necesita ser recreado.

La solución correcta es simplemente continuar con la actualización de rosdep:

bash 
rosdep update

¿Qué es rosdep?

rosdep es una herramienta de resolución de dependencias de ROS 2. Cuando un paquete de ROS 2 declara dependencias en su archivo package.xml (por ejemplo, <depend>rclpy</depend>), rosdep traduce esas claves abstractas en nombres de paquetes específicos del sistema operativo (por ejemplo, python3-rclpy en Ubuntu) y los instala automáticamente.

rosdep init crea el archivo de fuentes (/etc/ros/rosdep/sources.list.d/20-default.list) que apunta al índice central de dependencias de ROS. rosdep update descarga ese índice a la caché local del usuario para poder realizar consultas sin conexión.

Paso 10

Probar ROS 2

Para verificar que ROS 2 está instalado y funcionando correctamente, ejecute la prueba clásica de comunicación Publisher–Subscriber entre dos nodos de demostración.

Terminal 1 — Talker (Publisher)

Abra una terminal de Ubuntu y ejecute:

terminal 1 
ros2 run demo_nodes_cpp talker

Verá un flujo continuo de mensajes:

salida 
[INFO] [1720000000.000000000] [talker]: Publishing: 'Hello World: 1'
[INFO] [1720000001.000000000] [talker]: Publishing: 'Hello World: 2'
[INFO] [1720000002.000000000] [talker]: Publishing: 'Hello World: 3'

Terminal 2 — Listener (Subscriber)

Abra una segunda terminal de Ubuntu y ejecute:

terminal 2 
ros2 run demo_nodes_py listener

El listener comenzará a recibir y mostrar los mensajes publicados por el talker:

salida 
[INFO] [1720000002.100000000] [listener]: I heard: [Hello World: 3]
[INFO] [1720000003.100000000] [listener]: I heard: [Hello World: 4]
[INFO] [1720000004.100000000] [listener]: I heard: [Hello World: 5]

Conceptos: Publisher y Subscriber

Este ejemplo demuestra el patrón fundamental de comunicación en ROS 2: Publish/Subscribe.

  • Publisher (Talker) — Un nodo que publica mensajes en un tópico. En este caso, el nodo talker (escrito en C++) publica mensajes de tipo std_msgs/msg/String en el tópico /chatter.
  • Subscriber (Listener) — Un nodo que se suscribe a un tópico para recibir mensajes. El nodo listener (escrito en Python) se suscribe al tópico /chatter y ejecuta una función callback cada vez que recibe un mensaje.
  • Tópico — Un canal de comunicación con nombre (/chatter) donde los publishers envían datos y los subscribers los reciben. Los nodos no se comunican directamente entre sí; el tópico actúa como intermediario desacoplado.

Observe que el talker está escrito en C++ y el listener en Python. Esto demuestra que ROS 2 permite la comunicación transparente entre nodos de diferentes lenguajes de programación, gracias al middleware DDS.

Detener los nodos

Para detener cualquier nodo en ejecución, presione Ctrl + C en la terminal correspondiente.

Paso 12

Crear Workspace

Un workspace (espacio de trabajo) es un directorio donde se compilan y organizan los paquetes de ROS 2. Siga estos pasos para crear su primer workspace:

bash 
mkdir -p ~/ros2_ws/src
cd ~/ros2_ws
colcon build

Agregar el workspace al .bashrc

bash 
echo "source ~/ros2_ws/install/setup.bash" >> ~/.bashrc
source ~/.bashrc

Estructura de carpetas del workspace

Después de ejecutar colcon build, el workspace contiene cuatro directorios principales:

DirectorioDescripción
src/ Código fuente. Aquí se colocan los paquetes de ROS 2 (manualmente o clonados con git). Es el único directorio que el desarrollador modifica directamente.
build/ Artefactos de compilación. Contiene los archivos intermedios generados durante la compilación (archivos objeto, CMake cache, etc.). Se puede eliminar y regenerar con colcon build.
install/ Resultado de la instalación. Contiene los ejecutables, bibliotecas y archivos de configuración instalados. El archivo install/setup.bash hace que estos paquetes estén disponibles para ros2 run.
log/ Registros de compilación. Contiene los archivos de log generados por colcon durante cada compilación. Útil para depurar errores de compilación.
Limpieza

Si necesita una compilación limpia, elimine los directorios build/, install/ y log/, y ejecute colcon build nuevamente. Nunca elimine src/.

Paso 13

Crear un paquete Python

bash 
cd ~/ros2_ws/src

ros2 pkg create \
  --build-type ament_python \
  robotica_basica

Compile el workspace incluyendo el nuevo paquete:

bash 
cd ~/ros2_ws
colcon build --symlink-install

¿Qué es ament_python?

ament_python es el sistema de compilación de ROS 2 para paquetes escritos exclusivamente en Python. Es la contraparte de ament_cmake (para C++) dentro del ecosistema de compilación ament de ROS 2.

Cuando se crea un paquete con --build-type ament_python, se genera la siguiente estructura:

  • package.xml — Manifiesto del paquete con metadatos (nombre, versión, dependencias).
  • setup.py — Script de instalación de Python (compatible con setuptools).
  • setup.cfg — Configuración de instalación (install-scripts y script-dir).
  • robotica_basica/ — Directorio del módulo Python con un archivo __init__.py.
  • resource/ — Directorio de marcador para el índice de paquetes de ament.

La opción --symlink-install crea enlaces simbólicos en lugar de copiar archivos durante la instalación. Esto permite que los cambios en el código Python se reflejen inmediatamente sin necesidad de recompilar, acelerando significativamente el ciclo de desarrollo.

Paso 14

Ejecutar RViz2

bash 
rviz2

RViz2 (ROS Visualization 2) es la herramienta principal de visualización de ROS 2. Permite ver datos de sensores, modelos de robots (URDF), mapas, trayectorias, nubes de puntos y otros tipos de datos robóticos en un entorno 3D interactivo.

Advertencia esperada: Stereo is NOT SUPPORTED

Al iniciar RViz2 en WSL2, es común ver el siguiente mensaje en la terminal:

[INFO] [rviz2]: Stereo is NOT SUPPORTED
[INFO] [rviz2]: OpenGL version: 4.3 (Compatibility Profile) ...

Este mensaje no es un error. Indica que la renderización estereoscópica (para visores 3D o gafas de realidad virtual) no está disponible en el entorno WSLg. La funcionalidad de RViz2 no se ve afectada en absoluto. Todas las visualizaciones 2D y 3D funcionan normalmente.

Paso 15

Instalar Gazebo Harmonic

Gazebo Harmonic (también conocido como Gazebo Sim, antes Ignition Gazebo) es el simulador 3D de robótica recomendado para ROS 2 Jazzy. Permite simular robots, sensores y entornos físicos con un motor de física realista.

Instalación

bash 
sudo apt update

sudo apt install -y \
  ros-jazzy-ros-gz \
  ros-jazzy-ros-gz-sim \
  ros-jazzy-ros-gz-bridge \
  ros-jazzy-gz-tools-vendor
PaqueteFunción
ros-jazzy-ros-gzMetapaquete que agrupa todos los componentes de integración entre ROS 2 y Gazebo.
ros-jazzy-ros-gz-simPaquete de simulación que incluye el simulador Gazebo y archivos de launch para ROS 2.
ros-jazzy-ros-gz-bridgePuente bidireccional entre tópicos de ROS 2 y tópicos de Gazebo Transport. Permite que los datos del simulador fluyan hacia nodos de ROS 2 y viceversa.
ros-jazzy-gz-tools-vendorProporciona herramientas de línea de comandos para Gazebo (gz).

Verificar la instalación

bash 
ros2 pkg list | grep ros_gz

Deberá ver al menos los siguientes paquetes:

salida 
ros_gz
ros_gz_bridge
ros_gz_image
ros_gz_interfaces
ros_gz_sim

Problema conocido: gz sim — command not found

Comando gz sim no encontrado

En esta instalación, al intentar ejecutar Gazebo directamente con el comando gz sim, se obtuvo el error:

gz sim
command not found

Esto ocurre porque la instalación a través de los paquetes ros-jazzy-ros-gz-* no siempre coloca el ejecutable gz en el PATH del sistema. El binario está empaquetado como un vendor package y su ruta no se exporta globalmente.

La forma correcta y recomendada de ejecutar Gazebo Harmonic en esta configuración es a través de ROS 2 launch:

bash 
ros2 launch ros_gz_sim gz_sim.launch.py

Este comando inicia Gazebo Harmonic con un mundo vacío. Para abrir un mundo de ejemplo con formas geométricas y física activa, ejecute:

bash 
ros2 launch ros_gz_sim gz_sim.launch.py gz_args:="shapes.sdf"
Primer inicio

La primera vez que se ejecuta Gazebo, puede tardar varios minutos en cargar mientras descarga modelos y texturas. Las ejecuciones posteriores serán significativamente más rápidas porque los recursos se almacenan en caché localmente en ~/.gz/.

Problemas encontrados durante la instalación

La siguiente tabla documenta los problemas reales encontrados durante esta instalación y sus soluciones verificadas:

ProblemaCausaSolución
ERROR: default sources list already exists rosdep init ya fue ejecutado previamente (posiblemente durante la instalación de python3-rosdep). Ignorar el error y ejecutar rosdep update directamente.
No executable found al usar ros2 run El entorno de ROS 2 no ha sido cargado en la terminal actual. Falta ejecutar source. Ejecutar source /opt/ros/jazzy/setup.bash o abrir una nueva terminal (si ya se configuró .bashrc).
PackageNotFoundError al compilar El paquete no existe en el workspace, o el setup.bash del workspace no ha sido cargado. Verificar que el paquete existe en ~/ros2_ws/src/ y ejecutar source ~/ros2_ws/install/setup.bash.
BrokenPipeError al ejecutar ros2 topic list Error intermitente relacionado con la comunicación DDS cuando la red no está completamente inicializada en WSL2. Esperar unos segundos y reintentar. Si persiste, reiniciar WSL con wsl --shutdown desde PowerShell.
Stereo is NOT SUPPORTED WSLg no soporta renderización estereoscópica para visores 3D/VR. Ignorar el mensaje. No afecta la funcionalidad de RViz2.
gz sim — command not found El ejecutable gz no está en el PATH cuando se instala a través de los paquetes vendor de ROS 2. Utilizar ros2 launch ros_gz_sim gz_sim.launch.py en lugar del comando gz directo.

Buenas prácticas

Actualice frecuentemente

Ejecute sudo apt update && sudo apt upgrade -y al menos una vez por semana para mantener los paquetes de ROS 2 y del sistema operativo al día con las correcciones de seguridad y bugs.

No mezcle distribuciones de ROS 2

Evite tener paquetes de múltiples distribuciones (por ejemplo, Humble y Jazzy) en el mismo .bashrc. Solo haga source de una distribución a la vez.

Use --symlink-install para paquetes Python

Compile siempre con colcon build --symlink-install durante el desarrollo de paquetes Python. Esto evita recompilar después de cada cambio en el código fuente.

No edite archivos fuera de src/

Los directorios build/, install/ y log/ son generados automáticamente por colcon. Nunca modifique archivos en estos directorios manualmente.

Haga backup de .bashrc

Antes de modificar ~/.bashrc, haga una copia: cp ~/.bashrc ~/.bashrc.bak. Esto le permitirá restaurar la configuración si comete un error.

Reinicie WSL si experimenta problemas gráficos

Si RViz2 o Gazebo no se abren correctamente, ejecute wsl --shutdown desde PowerShell y vuelva a abrir Ubuntu. Esto reinicia el compositor WSLg y resuelve la mayoría de los problemas gráficos.

Use Git desde el primer momento

Inicialice un repositorio Git en su workspace: cd ~/ros2_ws/src && git init. Esto le permitirá rastrear cambios, revertir errores y colaborar con otros desarrolladores.

Cuestionario

1. ¿Qué kernel utiliza WSL2 para ejecutar distribuciones Linux?

  • Un kernel Windows NT traducido
  • Un kernel Linux real ejecutado sobre Hyper-V
  • Un kernel FreeBSD adaptado
  • El mismo kernel de Windows con una capa de compatibilidad

Respuesta: b) WSL2 ejecuta un kernel Linux real proporcionado y mantenido por Microsoft, dentro de una máquina virtual ligera gestionada por Hyper-V.

2. ¿Cuál es el codename de Ubuntu 24.04 LTS?

  • Jammy Jellyfish
  • Noble Numbat
  • Focal Fossa
  • Kinetic Kudu

Respuesta: b) Ubuntu 24.04 LTS tiene el codename Noble Numbat.

3. ¿Qué herramienta reemplaza a catkin_make en ROS 2?

  • cmake
  • make
  • colcon
  • rosdep

Respuesta: c) colcon (collective construction) es el sistema de compilación que reemplaza a catkin_make de ROS 1.

4. ¿Por qué ROS 2 requiere que el locale del sistema esté configurado en UTF-8?

  • Para soportar nombres de archivos con acentos
  • Porque DDS utiliza UTF-8 como codificación estándar para mensajes
  • Para compatibilidad con Python 2
  • Es un requisito del sistema de compilación CMake

Respuesta: b) El middleware DDS de ROS 2 utiliza UTF-8 para la serialización de mensajes y la interoperabilidad entre nodos.

5. ¿Qué hace el archivo /opt/ros/jazzy/setup.bash?

  • Instala ROS 2 Jazzy
  • Compila todos los paquetes del workspace
  • Configura las variables de entorno necesarias para usar ROS 2
  • Actualiza los repositorios de paquetes

Respuesta: c) El archivo setup.bash configura las variables PATH, LD_LIBRARY_PATH, PYTHONPATH, AMENT_PREFIX_PATH y ROS_DISTRO.

6. ¿Cuál es la diferencia principal entre ros-jazzy-ros-base y ros-jazzy-desktop?

  • ros-base incluye Gazebo y desktop no
  • desktop incluye RViz2 y herramientas gráficas; ros-base solo incluye las bibliotecas de comunicación
  • Son exactamente iguales
  • ros-base solo funciona en ARM y desktop solo en x86

Respuesta: b) desktop incluye todo lo de ros-base más RViz2, rqt, nodos de demostración y herramientas de visualización.

7. ¿Qué ocurre si se usa > en lugar de >> al modificar .bashrc?

  • Se agrega la línea al final del archivo
  • Se sobrescribe todo el contenido del archivo
  • Se crea un archivo de respaldo automático
  • No hay diferencia

Respuesta: b) El operador > sobrescribe el archivo completo, mientras que >> agrega al final (append).

8. ¿Qué directorio del workspace contiene el código fuente de los paquetes?

  • build/
  • install/
  • src/
  • log/

Respuesta: c) El directorio src/ es donde se colocan los paquetes de ROS 2 con su código fuente.

9. ¿Qué hace rosdep?

  • Compila paquetes de ROS 2
  • Resuelve e instala las dependencias del sistema declaradas en package.xml
  • Publica mensajes en tópicos
  • Configura la red para DDS

Respuesta: b) rosdep traduce las dependencias abstractas de package.xml a paquetes del sistema operativo y los instala.

10. ¿Qué ventaja ofrece --symlink-install durante la compilación?

  • Compila más rápido usando múltiples núcleos
  • Crea enlaces simbólicos para que los cambios en Python se reflejen sin recompilar
  • Instala los paquetes en /opt/ros/ en lugar del workspace
  • Habilita la depuración paso a paso

Respuesta: b) --symlink-install crea enlaces simbólicos desde install/ hacia src/, permitiendo que los cambios en scripts Python estén disponibles inmediatamente.

11. ¿Qué componente de WSL2 permite ejecutar aplicaciones gráficas de Linux en Windows?

  • VcXsrv
  • X11
  • WSLg
  • Remote Desktop

Respuesta: c) WSLg (Windows Subsystem for Linux GUI) proporciona soporte nativo para aplicaciones gráficas de Linux sin necesidad de un servidor X externo.

12. ¿Cuál es el patrón de comunicación demostrado por los nodos Talker y Listener?

  • Request/Response
  • Publish/Subscribe
  • Action Client/Server
  • Parameter Server

Respuesta: b) El patrón Publish/Subscribe es el modelo fundamental de comunicación en ROS 2, donde un publisher envía datos a un tópico y un subscriber los recibe.

13. ¿Qué sistema de compilación se usa para paquetes Python puros en ROS 2?

  • ament_cmake
  • catkin
  • ament_python
  • setuptools

Respuesta: c) ament_python es el sistema de compilación de ROS 2 específico para paquetes escritos exclusivamente en Python.

14. ¿Cómo se ejecuta Gazebo Harmonic cuando el comando gz sim no funciona en la instalación a través de paquetes ROS?

  • gazebo
  • ignition gazebo
  • ros2 launch ros_gz_sim gz_sim.launch.py
  • roslaunch gazebo_ros gazebo.launch

Respuesta: c) La forma recomendada es utilizar el archivo de launch de ROS 2: ros2 launch ros_gz_sim gz_sim.launch.py.

15. ¿Cuál es el middleware de comunicación utilizado por ROS 2?

  • TCP/IP sockets
  • ZeroMQ
  • DDS (Data Distribution Service)
  • MQTT

Respuesta: c) ROS 2 utiliza DDS (Data Distribution Service) como middleware de comunicación, un estándar industrial de OMG que soporta comunicación en tiempo real, descubrimiento automático de nodos y QoS configurable.
